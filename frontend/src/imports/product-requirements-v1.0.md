## 1. 产品概述

### 1.1 背景与动机

大量企业知识沉淀在 PDF 报告、Word 文档、PPT 演示、Excel 数据表、扫描图片等非结构化文件中。传统的全文检索和关键词搜索无法理解语义，难以跨文档综合推理，导致知识获取效率低下。

基于大语言模型（LLM）的 RAG（检索增强生成）技术能够弥补上述缺陷，但现有方案普遍存在：

- 需要开发者手动配置文档解析、索引构建流程，非技术用户无法使用
- 单一向量检索对"谁是什么、有什么属性"等实体类问题效果差
- AI 回答缺乏透明来源，用户无法核实
- 长耗时处理任务（解析/建索引）缺少可视化进度，用户体验差

GraphRAG Agent 正是为解决上述问题而生。

### 1.2 核心目标

| 目标 | 衡量指标 |
|------|---------|
| 多格式文档一键接入 | 支持 PDF/Word/PPT/Excel/图片/HTML，上传后全自动处理，用户零配置 |
| 双引擎知识索引 | 向量检索（语义相似度）+ 知识图谱检索（实体属性），覆盖更广泛问题类型 |
| 答案有据可查 | 每条 AI 回答均展示来源：KG 实体卡 + 原文段落双路溯源 |
| AI 推理透明 | 展示路由策略、问题改写次数、检索充分性、推理耗时 |
| 处理过程可视 | 文档状态从上传到就绪，全程进度可见，无黑盒等待 |

### 1.3 业务痛点

**痛点 1：文档处理门槛高**  
现有 RAG 工具需要用户自行安装 Python 环境、执行命令行脚本，非技术用户完全无法使用。本产品将全部流程封装为 Web 界面，上传即自动触发解析和索引构建。

**痛点 2：单一检索效果不稳定**  
纯向量检索在"某技术的核心特点是什么"等实体属性类问题上召回率低；纯关键词检索则在"功能上 A 和 B 有什么异同"等跨段落综合推理时效果差。本产品采用向量 + BM25 + 知识图谱三路融合，覆盖全类型问题。

**痛点 3：AI 回答无溯源，不可信**  
用户无法判断 AI 回答是来自文档还是凭空生成。本产品每次回答都附带精确的来源：命中的知识图谱实体（含原文片段）和向量检索到的原始段落。

**痛点 4：长耗时任务无反馈**  
大型 PDF 的 MinerU 解析可能需要数分钟，建索引也需要时间，用户在黑盒等待中容易放弃。本产品提供全程状态机进度展示，消除等待焦虑。

**痛点 5：多文档知识库管理混乱**  
企业用户需要管理数十个文档，需要清晰的列表视图、状态过滤、文档删除等管理功能。

### 1.4 目标用户

| 用户类型 | 特征描述 | 典型使用场景 |
|---------|---------|------------|
| **知识工作者（核心用户）** | 企业中层，使用 Word/PPT/Excel，非技术背景 | 上传季报/行业报告，提问"Q1 销售额同比增长多少" |
| **研究员 / 分析师** | 需要快速消化大量文献、报告 | 上传多篇学术 PDF，提问"这几篇文章中关于 X 技术的核心观点" |
| **产品经理 / 运营** | 需要调取文档中的具体数据和细节 | 上传产品文档，提问"当前版本支持哪些文件格式" |
| **技术评估者（次要）** | 评估 RAG 系统能力的技术人员 | 关注路由策略、检索充分性、KG 实体质量等系统指标 |

### 1.5 核心价值主张

```
上传 → 等待 → 提问 — 三步核心流程，无需任何技术配置
答案可溯源 — KG 实体 + 原文段落，每条回答有据可查
过程透明 — 文档处理进度 + AI 推理路径，完全可见
```

## 2. 产品范围与边界

### MVP 范围内（本期交付）

- 文件上传（支持 PDF/Word/PPT/Excel/图片/HTML）
- 文档解析（MinerU Cloud API 自动触发）
- 双路索引构建（向量索引 + 知识图谱索引）
- 文档状态全程可视化（轮询更新）
- 单文档 / 多文档范围内的自然语言问答
- 结构化来源溯源（KG 实体卡 + 原文段落）
- AI 推理元信息展示（路由/改写/充分性/耗时）
- 文档管理（列表 / 删除）
- 系统健康状态监控

### MVP 范围外（后续迭代）

- 多轮对话（会话记忆，当前 session_id 预留但未实现）
- 流式输出（SSE streaming，当前同步返回）
- 用户账户系统与多租户
- 文档共享与权限控制
- 基于 Excel 的 KG 抽取（当前 Agent API 仅输出 Markdown，无结构化 JSON）
- 文档重新索引（INDEX_FAILED 后手动重试）

## 3. 完整产品流程

### 3.1 主流程总览

```
用户上传文档
     │
     ▼
[MinerU 云端解析]  ←── 自动触发，无需手动操作
     │
     ▼
[双路索引构建]  ←── 解析完成后自动触发
  ├── 向量索引（Chroma + BM25 混合检索）
  └── 知识图谱索引（LangExtract 实体抽取）
     │
     ▼
文档状态 = READY
     │
     ▼
用户输入问题
     │
     ▼
[Agentic RAG 问答]
  ├── 路由分类（4种策略）
  ├── 双路检索（KG + 向量）
  ├── 检索质量评估（必要时改写重检）
  └── LLM 生成答案
     │
     ▼
返回答案 + 来源溯源
```

### 3.2 文件上传与解析流程

**入口**：知识库管理页（`/knowledge`）上传区

```
步骤 1：用户选择文件
  触发方式：拖拽到上传区 / 点击"选择文件"按钮
  前端校验：
    - 格式检验：pdf/doc/docx/ppt/pptx/xls/xlsx/jpg/jpeg/png/gif/webp/html
    - 大小校验：Excel ≤ 10MB；其他 ≤ 200MB
    - 校验失败：Toast 错误提示，不发起请求
  界面变化：显示文件名 + 大小 + [上传并解析] 按钮激活

步骤 2：提交上传
  请求：POST /api/v1/documents/upload（multipart/form-data）
  参数：file（文件二进制）+ enable_ocr（默认 true）
  过程输出：
    - HTTP 202 响应，获取 doc_id
    - 界面：进度条显示上传进度（HTTP XHR progress 事件）
    - 文档条目出现在左侧列表，状态 = PARSING

步骤 3：MinerU 解析（异步，后端自动执行）
  路由逻辑（后端）：
    Excel（.xls/.xlsx） → MinerU Agent API（任务制，返回 task_id）
    其他所有格式       → MinerU Precision API（批量，返回 batch_id）
  后端轮询 MinerU 状态（每 10s，最多 600s）
  解析成功：
    - 下载 ZIP → 解压到 mineru_parser/output/{doc_id}/
    - 文档状态更新 → PARSED
    - 自动触发索引构建
  解析失败：
    - 文档状态更新 → PARSE_FAILED，记录 error_msg
    - 前端展示失败原因 + [重新上传] 按钮

过程输出物（后端）：
  Precision API：full.md + content_list.json + images/
  Agent API：   full.md（仅 Markdown）
```

### 3.3 索引构建流程

**触发条件**：MinerU 解析完成（status = PARSED）后自动触发

```
两路索引并行构建：

路径 A — 向量索引
  执行：python rag_pipeline/pipeline.py build {doc_id}
  读取：full.md（Markdown 全文）
  处理：文档分块 → 向量化（Embedding）→ 写入 Chroma DB + BM25 索引
  完成：vector_status = done，记录 chunk_count
  失败：vector_status = failed

路径 B — 知识图谱索引
  执行：python langextract_pipeline/run_pipeline.py "{doc_id}"
  读取：content_list.json（结构化内容列表）
  处理：逐段 LLM 实体抽取 → 去重 → 写入 .jsonl 文件
  完成：kg_status = done，统计 entity_count（match_exact 类型）
  失败：kg_status = failed

状态判定：
  两路均 done → 文档 status = READY
  任一路 failed → 文档 status = INDEX_FAILED

过程输出物：
  向量索引：chroma_db/ 中对应 collection
  KG 索引：langextract_pipeline/output/{doc_id}.jsonl
```

### 3.4 智能问答流程

**入口**：智能问答页（`/chat`）输入框

```
步骤 1：用户输入问题并发送
  触发方式：按 Enter / 点击发送按钮
  前端校验：
    - 问题为空 → 输入框 shake 动画，不发送
    - 无 READY 文档 → Toast "请先上传文档"，跳转知识库页
  输入框禁用，Chat Area 追加 AI 回答卡片（loading 状态）

步骤 2：后端接收问题
  请求：POST /api/v1/qa/query
  参数：
    question（必填）：用户问题，1~500 字符
    doc_ids（可选）：限定检索范围；null = 全库检索
    session_id（可选）：预留，当前不生效

步骤 3：Agentic RAG 执行（后端，LangGraph 5节点）

  节点 1 — route_question（路由分类）
    输入：用户问题
    执行：LLM 判断问题类型
    输出：route 字段（4 种策略之一）
    ┌──────────────┬──────────────────────────────┐
    │ route        │ 适用场景                      │
    ├──────────────┼──────────────────────────────┤
    │ entity_query │ "X 是什么/有什么属性"          │
    │ semantic_q.  │ "XX 如何工作/解释概念"        │
    │ hybrid_query │ "X 和 Y 的区别/综合比较"      │
    │ direct_answer│ 通用知识，无需检索             │
    └──────────────┴──────────────────────────────┘

  节点 2 — retrieve（双路检索）
    KG 检索：BM25 过 JSONL 文件，返回 Top-5 实体（含属性 + 原文片段）
    向量检索：Chroma 向量 0.6 + BM25 0.4 混合 → RRF 融合 → Top-5 段落
    输出：kg_results（实体列表）+ passage_results（段落列表）

  节点 3 — grade_documents（检索评估）
    输入：问题 + 双路检索结果
    执行：LLM 判断检索内容是否足以回答问题
    输出：sufficient（布尔值）
    决策：sufficient=false 且 rewrite_count < MAX_REWRITE → 触发改写
           sufficient=true  → 直接生成答案

  节点 4 — rewrite_question（问题改写，条件触发）
    执行：LLM 改写问题，rewrite_count + 1
    输出：新问题 → 重新进入 retrieve 节点
    限制：最多改写 2 次

  节点 5 — generate_answer（答案生成）
    输入：问题 + merged_context（KG 实体 + 段落合并）
    执行：LLM 综合推理生成答案
    输出：Markdown 格式答案（含引用标注）

步骤 4：返回结构化响应
  answer：Markdown 格式答案
  meta：{ route, rewrite_count, question_used, original_question, sufficient, latency_ms }
  sources：{ kg_entities: [...], passages: [...] }

步骤 5：前端渲染
  answer → react-markdown 渲染（支持表格/代码块/粗体等 GFM 语法）
  meta → 底部 Tag 行（路由 + 改写次数 + 充分性 + 耗时）
  sources → 右侧 Source Panel（KG 实体卡 + 原文段落卡）
  右侧面板自动展开（若折叠状态）
  输入框恢复可用
```

### 3.5 错误处理流程

| 错误场景 | 触发条件 | 前端处理 | 后端状态 |
|---------|---------|---------|---------|
| 格式不支持 | 上传 .mp4 等非支持格式 | 即时校验，Toast 红色提示，不发请求 | — |
| 文件过大 | Excel > 10MB 或其他 > 200MB | 即时校验，Toast 提示限制 | — |
| MinerU 解析超时 | 600s 内未完成 | 文档条目红色 + 错误原因 + [重新上传] | PARSE_FAILED |
| 索引构建失败 | subprocess 非 0 退出 | 文档条目红色 + 错误原因 | INDEX_FAILED |
| 文档未就绪提问 | doc_ids 中含非 READY 文档 | Toast "文档处理中，请等待" | 400 DOCUMENT_NOT_READY |
| 问答超时 | 后端 60s 无响应 | Toast "问答超时" + [重新发送] 按钮 | 504 |
| 网络断开 | 任意请求失败 | Toast + 自动重试 3 次 | — |
| 处理中删除 | 删除 PARSING/INDEXING 文档 | 提示"处理中不可删除" | 400 DOCUMENT_BUSY |

## 6. 核心功能交互逻辑

### 6.1 文件上传交互

```
用户拖入或选择文件
        │
        ▼
前端即时校验
  ├── 格式不支持 → Toast 错误（红色）+ 清空选区
  ├── 文件过大   → Toast 错误（红色）+ 清空选区
  └── 通过校验   → 显示文件预览条 + [上传并解析] 按钮激活
        │
        ▼ 用户点击 [上传并解析]
进度条显示（XHR progress 事件）
POST /api/v1/documents/upload
        │
  ┌─────┴────────┐
  │ 成功（202）  │ 失败（400/413）
  │              ▼
  │     Toast 错误 + 重置上传区
  ▼
获取 doc_id
文档条目追加至列表（status=PARSING，蓝色）
上传区重置为空闲状态
启动轮询 setInterval(5000)
```

### 6.2 文档状态轮询与进度展示

**轮询策略**：
- 轮询间隔：5 秒/次
- 触发条件：文档 status 不是 READY / PARSE_FAILED / INDEX_FAILED
- 停止条件：到达终止状态（READY 或 FAILED）
- 最大轮询时间：前端保护性超时 700s（略大于后端 600s MinerU 超时）

**进度条计算规则**：

| 状态 | 进度百分比 |
|------|----------|
| UPLOADED | 5% |
| PARSING | 5%~50%（基于已等待时间线性插值，最多 50%） |
| PARSED | 55% |
| INDEXING | `55% + vector_progress × 22.5% + kg_progress × 22.5%` |
| READY | 100% |

其中：`pending=0%`，`building=50%`（无精确进度时的估算），`done=100%`

**状态变化的视觉反馈**：
- PARSING → PARSED：进度条跳至 55%，Toast "文档解析完成，开始构建索引"
- INDEXING → READY：进度条跳至 100%（绿色），Toast "文档已就绪，可以提问"（带 [立即提问] 跳转按钮）
- 任何 FAILED：进度条变红，Toast 显示错误原因

### 6.3 问答交互

```
用户输入问题 → Enter / 点击发送
        │
        ▼

```

### 8.2 关键组件行为说明

**UploadZone**：
- 支持 `dragenter/dragover/dragleave/drop` 事件
- `dragover` 时：边框变为 Accent Purple + 背景微亮
- 文件落下时：立即校验格式和大小
- 内部维护状态：`idle / dragover / selected / uploading`

**DocumentListItem**：
- 非终止状态（PARSING/INDEXING）：每 5s 更新进度展示（来自轮询）
- 点击条目：在主区域展示 DocumentDetail
- 长按/右键：弹出操作菜单（仅 READY/FAILED 状态可删除）

**AIMessage**：
- 渲染 `react-markdown` + `remark-gfm`（支持表格/代码块/加粗/斜体）
- 底部 Tag 行使用 RouteTag、StatusDot 等复合组件
- "查看来源" 按钮点击：通过 Zustand 触发 SourcePanel 展开

**KGEntityCard**：
- 属性 key-value 列表，超过 4 个属性时折叠显示
- `context_snippet` 默认不显示，点击"展开原文片段"后展示（最多 200 字符）
- 实体 type 对应 Badge 颜色：使用 Accent Purple（product/concept）/ Accent Blue（technology/framework）等

## 10. 数据接口规范（前后端对接）

### 10.1 API 总览

| 接口 | 方法 | 路径 | 功能 |
|------|------|------|------|
| 上传文档 | POST | `/api/v1/documents/upload` | 上传文件，触发异步解析 |
| 查询文档状态 | GET | `/api/v1/documents/{doc_id}/status` | 轮询文档处理进度 |
| 获取文档列表 | GET | `/api/v1/documents` | 列出所有文档及状态 |
| 删除文档 | DELETE | `/api/v1/documents/{doc_id}` | 删除文档及索引 |
| 智能问答 | POST | `/api/v1/qa/query` | 提问，返回答案+来源 |
| 健康检查 | GET | `/api/v1/health` | 系统组件状态 |

### 10.2 关键响应字段映射（QAResponse → 前端 UI）

| 后端字段路径 | 前端 UI 元素 |
|------------|------------|
| `answer` | AIMessage 内容区（react-markdown 渲染）|
| `meta.route` | RouteTag 组件（路由策略标签，颜色见7.7节）|
| `meta.rewrite_count` | 改写次数 Tag（0次灰色，>0黄色）|
| `meta.sufficient` | 充分性 Tag（true=绿勾，false=黄警告）|
| `meta.latency_ms` | 耗时 Tag（毫秒转秒显示）|
| `sources.kg_entities[].name` | KGEntityCard 标题 |
| `sources.kg_entities[].type` | KGEntityCard type Badge |
| `sources.kg_entities[].attributes` | KGEntityCard 属性 key-value 列表 |
| `sources.kg_entities[].context_snippet` | KGEntityCard 折叠原文片段 |
| `sources.kg_entities[].document_id` | KGEntityCard 来源文档标注 |
| `sources.passages[].content` | PassageCard 正文 |
| `sources.passages[].section` | PassageCard 章节标题 |
| `sources.passages[].page` | PassageCard 页码标注 |
| `sources.passages[].chunk_type` | PassageCard type Badge（text/table）|
| `sources.passages[].entities` | PassageCard 实体 Tag 列表 |

### 10.3 文档状态映射（DocumentStatus → 前端 UI）

| 后端 status 字段 | 前端进度显示 | 徽章样式 |
|----------------|------------|---------|
| `UPLOADED` | 5% | Info Blue + ↑ |
| `PARSING` | 5%~50%（时间插值）| Info Blue + ⟳ 旋转 |
| `PARSED` | 55% | Warning + ⟳ 旋转 |
| `INDEXING` | 55%~100%（双路进度）| Warning + ⟳ 旋转 |
| `READY` | 100% | Success Green + ● 发光 |
| `PARSE_FAILED` | — （隐藏进度条）| Error Red + ⊗ |
| `INDEX_FAILED` | — （隐藏进度条）| Error Red + ⊗ |

### 10.4 错误码处理规范

| 错误码 | HTTP 状态 | 前端处理 |
|--------|---------|---------|
| `UNSUPPORTED_FORMAT` | 400 | Toast 红色："不支持 .xxx 格式" |
| `FILE_TOO_LARGE` | 413 | Toast 红色："文件超过限制" |
| `DOCUMENT_NOT_FOUND` | 404 | Toast + 刷新列表 |
| `DOCUMENT_BUSY` | 400 | Toast："文档处理中，请等待完成" |
| `DOCUMENT_NOT_READY` | 400 | Toast："文档索引中，请稍候" |
| `INVALID_QUESTION` | 400 | 输入框 shake 动画 + 红色提示 |

## 11. 非功能性需求

### 11.1 性能

| 指标 | 要求 |
|------|------|
| 页面首屏加载 | ≤ 2s（Vite 产物，Gzip 压缩后） |
| 轮询响应 | 状态接口 P99 ≤ 500ms |
| 问答响应 | P50 ≤ 5s；P95 ≤ 15s；超过 60s 返回 504 |
| 文档列表渲染 | 100 条文档虚拟滚动，不卡顿 |
| Markdown 渲染 | 3000 字符以内 ≤ 100ms |

### 11.2 可靠性

| 场景 | 要求 |
|------|------|
| 前端网络断开 | Toast 提示 + 指数退避重试（3次）|
| 后端服务重启 | 前端自动重连，轮询不丢失 |
| 大文件上传中网络断开 | Toast 错误 + 允许重新上传（不自动续传，MVP）|
| 浏览器刷新 | Zustand 状态持久化（localStorage），恢复文档列表 |

### 11.3 可访问性（a11y）

- 所有交互元素支持键盘操作（Tab 焦点顺序合理）
- 状态徽章不纯依赖颜色（同时有文字和图标）
- 图标按钮均有 aria-label
- 错误提示不仅 Toast，表单字段内联错误文字

### 11.4 浏览器兼容性

| 浏览器 | 最低版本 |
|--------|---------|
| Chrome | 100+ |
| Firefox | 100+ |
| Safari | 15+ |
| Edge | 100+ |
| 移动端 Safari | iOS 15+ |
